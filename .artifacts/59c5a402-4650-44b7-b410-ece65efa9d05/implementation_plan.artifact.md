# Fix Metadata Assignment and Display (Moods, Tags, Instruments)

The user reported that adding songs to Moods, Tags, or Instruments is not working. The investigation revealed a mismatch between how songs are identified in the metadata system and the rest of the app, as well as several logic errors in the display and assignment code.

## Analysis of the Problem

1.  **ID Mismatch & Constraint Failures**:
    -   The app identifies songs using **MediaStore IDs** (`Song.id`).
    -   However, metadata relationship tables (e.g., `song_mood_relationship`) have foreign keys pointing to `SongEntity(song_key)`.
    -   `SongEntity` is a Room table primarily used for **playlists**, and `song_key` is an autogenerated primary key.
    -   If a song is not in a playlist, it doesn't exist in `SongEntity`. Even if it does, its `song_key` is different from its MediaStore ID.
    -   This causes assignment insertions to either fail (due to foreign key violations) or link to the wrong records.

2.  **Display Logic**:
    -   Mood/Tag/Instrument detail screens only show "Top Songs" fetched from ranking tables.
    -   Ranking tables only contain songs with play history. Newly added songs are missing from these tables.
    -   The detail screens do not query the relationship tables to show all assigned songs.

3.  **ViewModel Bug**:
    -   `MoodDetailViewModel.assignSongs` has swapped arguments when calling the repository, effectively trying to assign "Mood IDs" to "Song IDs" in reverse.

## Proposed Changes

### 1. Database Schema Refactor

-   **Standardize on MediaStore ID**: All metadata relationship and ranking tables will use the MediaStore ID as `song_id`.
-   **Remove Foreign Keys to `SongEntity`**: Since `SongEntity` is playlist-specific, metadata should not depend on it. We will remove the foreign keys to `SongEntity(song_key)` from:
    -   `SongMoodEntity`, `SongTagEntity`, `SongInstrumentEntity`, `SongGenreEntity`.
    -   `MoodSongRankingEntity`, `TagSongRankingEntity`, `InstrumentSongRankingEntity`, `GenreSongRankingEntity`.
-   **Note**: We will keep foreign keys to the metadata master tables (e.g., `moods(id)`) to ensure integrity when a mood is deleted.

### 2. Data Layer Updates

-   **MetadataDao**:
    -   Add `getSongIdsForMood`, `getSongIdsForTag`, and `getSongIdsForInstrument`.
    -   Update stats queries to ensure they correctly join and count based on MediaStore IDs.
-   **RankingDao**:
    -   Update queries to return `List<Long>` (song IDs) instead of `List<SongEntity>`.
-   **MetadataRepository**:
    -   Update `MoodDetail`, `TagDetail`, and `InstrumentDetail` data classes to include a `songs: List<Song>` field representing all assigned songs.
    -   Update `RealMetadataRepository` to fetch all song IDs for a given metadata item and convert them to `Song` objects using `SongRepository`.

### 3. UI and ViewModel Fixes

-   **MoodDetailViewModel**: Fix the argument swap in `assignSongs`.
-   **MoodDetailScreen** (and Tag/Instrument screens):
    -   Update the "Top Songs" section (or add a new "All Songs" section) to display the full list of assigned songs.
    -   Currently, it only takes the first 5 "Top Songs". We should either show more or provide a "See All" option.

## Verification Plan

### Automated Tests
-   Verify that assigning a song to a mood via `LibraryViewModel` result in a successful entry in `song_mood_relationship` with the correct MediaStore ID.
-   Verify that `getMoodDetail` returns the newly added song in its `songs` list.

### Manual Verification
1.  Open a song's context menu and select "Add to Mood".
2.  Choose a mood and apply.
3.  Go to the Moods library, open that mood, and verify the song is listed.
4.  Repeat for Tags and Instruments.
5.  Verify that songs not in any playlist can also be tagged successfully.
